## 그 스택이 강한 지점

- **일반 웹 서비스·대시보드·SaaS**: T3(Next.js+tRPC+TypeScript+Tailwind 계열) 자체가 모던 웹 앱용으로 설계돼서, CRUD·인증·멀티테넌트 같은 건 매우 잘 커버합니다.[dev+1](https://dev.to/4r7ur/unleash-the-potential-of-t3-stack-m4k)
    
- SSR/CSR/ISR 섞인 제품형 웹앱: SEO 필요한 마케팅 페이지 + 로그인 뒤 대시보드 구조에 특히 잘 맞습니다.[[aalpha](https://www.aalpha.net/articles/nextjs-advantages-and-disadvantages/)]​
    
- API 백엔드: Next.js API Routes/Route Handlers와 Railway(장기 실행 가능) 조합이면 평범한 REST/GraphQL/tRPC 서버는 무리 없습니다.[lazyadmin+1](https://lazyadmin.nl/it/railway-vs-vercel-choosing-the-right-hosting-for-your-app/)
    
- 배포 경험: Vercel은 프론트·서버리스에 최적화, Railway는 백엔드·잡·데이터베이스에 좋아서, 둘 합치면 대부분의 “일반 스타트업 웹 제품” 시나리오는 커버됩니다.[railway+2](https://docs.railway.com/maturity/compare-to-vercel)
    

현실적으로, “대부분의 웹 스타트업 MVP/프로덕트”를 이 조합 하나로 밀어붙여도 되는 수준이라는 점에서는 당신 말에 상당히 가까워요.[codingcops+1](https://codingcops.com/t3-stack/)

## 하지만 한계가 분명한 영역

아래 같은 건 그 스택만으로는 힘들거나, 억지로 하면 손해가 큽니다.

- 실시간·장기 연결(웹소켓, 게임 서버 등): Vercel 서버리스는 기본적으로 장기 연결과 상주 프로세스에 안 맞고, 런타임 제한과 콜드 스타트가 있어서 실시간 시스템에는 제약이 큽니다.[ritza+2](https://ritza.co/articles/gen-articles/cloud-hosting-providers/railway-vs-vercel/)
    
- 무거운 배치 잡·ML 추론: Vercel 함수는 메모리(4GB)·실행 시간(최대 800초) 제한이 있고, 대용량 처리나 GPU가 필요한 작업에는 구조적으로 불리합니다.[railway+1](https://docs.railway.com/maturity/compare-to-vercel)
    
- 엣지/고성능 특화 요구: T3에서 자주 쓰는 Prisma는 번들 크기와 러스트 바이너리 때문에 엣지 런타임에서 제약이 있고, 고성능 엣지 컴퓨트에는 적합하지 않다는 비판이 있습니다.[[youtube](https://www.youtube.com/watch?v=-R9uHm0G5m4)]​
    
- JS 생태계 밖 클라이언트: tRPC는 타입 세이프하고 좋지만, JS/TS 중심이라 다른 언어 클라이언트(모바일 네이티브, 일부 서버 언어) 지원은 상대적으로 약하다는 지적이 있습니다.[[redskydigital](https://redskydigital.com/why-the-t3-stack-shines-as-2025s-typescript-dream-team/)]​[[youtube](https://www.youtube.com/watch?v=-R9uHm0G5m4)]​
    
- 극단적으로 커진 코드베이스: tRPC 엔드포인트 수가 수백·수천 개로 가면 TS 서버 성능 저하, DX 저하 같은 이슈가 보고되어 있습니다.[[youtube](https://www.youtube.com/watch?v=-R9uHm0G5m4)]​
    

이런 것들은 “다른 도구를 추가로 쓰거나, 아예 다른 런타임/플랫폼을 병행해야 하는 영역”이라서, “이 스택만으로 상상 가능한 거의 모든 작업”이라는 표현과는 거리가 있습니다.[[docs.railway](https://docs.railway.com/maturity/compare-to-vercel)]​[[youtube](https://www.youtube.com/watch?v=-R9uHm0G5m4)]​

## Railway·Vercel 관점의 제약

- Vercel:
    
    - 장점: 정적/서버리스/프론트엔드 배포에 최적, 전 세계 CDN, 자동 스케일.[lazyadmin+1](https://lazyadmin.nl/it/railway-vs-vercel-choosing-the-right-hosting-for-your-app/)
        
    - 단점: 함수당 메모리·시간 제한, 콜드 스타트, 장기 실행 잡/웹소켓/커스텀 런타임에 불리.[ritza+1](https://ritza.co/articles/gen-articles/cloud-hosting-providers/railway-vs-vercel/)
        
- Railway:
    
    - 장점: 항상 켜져 있는 백엔드, DB와 같은 프라이빗 네트워크, 언리미티드 런타임, 커스텀 Docker까지 지원해서 백엔드·잡에 훨씬 자유롭습니다.[lazyadmin+2](https://lazyadmin.nl/it/railway-vs-vercel-choosing-the-right-hosting-for-your-app/)
        
    - 그래도 한계: 결국 “클라우드 VM/컨테이너” 수준이라, 초대형 분산 시스템·전문 PaaS(예: GPU 클러스터, 대규모 데이터 레이크) 영역은 별도 인프라가 필요합니다.[lazyadmin+1](https://lazyadmin.nl/it/railway-vs-vercel-choosing-the-right-hosting-for-your-app/)
        

즉, 둘을 잘 섞으면 “일반적인 B2B/B2C 웹 서비스”는 충분하지만, 모든 인프라·컴퓨팅 시나리오를 포괄하지는 못합니다.[ritza+2](https://ritza.co/articles/gen-articles/cloud-hosting-providers/railway-vs-vercel/)

## T3 자체에 대한 비판 포인트

T3 저자 본인도 약점들을 꽤 솔직하게 인정합니다.[[youtube](https://www.youtube.com/watch?v=-R9uHm0G5m4)]​

- 모듈성 vs 프리셋: create-t3-app은 빠르게 시작하기 좋지만, 큰 팀·복잡한 요구에 맞춰 세밀하게 튜닝하려면 오히려 발목 잡힐 수 있다는 지적이 있습니다.[[dev](https://dev.to/4r7ur/unleash-the-potential-of-t3-stack-m4k)]​
    
- Next.js 의존성: 라우팅·데이터 로딩 방식이 Next 중심이라, 다른 런타임/프레임워크와 섞기엔 유연성이 떨어집니다.[edana+1](https://edana.ch/en/2025/10/09/next-js-strengths-and-limitations-of-the-full-stack-react-framework/)
    
- 타입 세이프의 비용: tRPC/TypeScript 전체 스택은 DX는 뛰어나지만, 매우 큰 코드베이스에선 타입체커 비용·복잡성이 커집니다.[[redskydigital](https://redskydigital.com/why-the-t3-stack-shines-as-2025s-typescript-dream-team/)]​[[youtube](https://www.youtube.com/watch?v=-R9uHm0G5m4)]​
    

그래서 “모던 타입세이프 웹 앱의 해피 패스”에는 최적인 대신, 그 바깥으로 나가면 금방 마찰이 생깁니다.[cubettech+1](https://cubettech.com/resources/blog/unveiling-the-T3-stack-powering-web-applications-with-technology-scalability-and-security/)
