# **5. 실제 MVP 개발에 바로 연결되는 질문**

현재 문서 기준으로 MVP는 **유사 사례 검색 → 작업지시서 초안 → 원가·마진 검토 → 누락 사양 질문 → 승인본 저장** 흐름이 가장 적합합니다. 이를 검증하려면 아래 질문이 중요합니다.

66. 제품 사진이나 설명만 받았을 때, 사람이 작업지시서 초안을 만들기 위해 추가로 묻는 질문은 무엇인가요?
67. 작업지시서 초안을 만들 때 최소 입력값은 무엇인가요?  
    예: IP명, 상품 종류, 사이즈, 수량, 소재, 색상, 납기, 판매처, 목표가.
68. AI가 초안을 만든 뒤 사람이 가장 먼저 검토할 부분은 어디인가요?
69. “누락 사양 질문”을 AI가 자동으로 한다면, 반드시 물어봐야 할 질문 리스트는 무엇인가요?
70. 유사 제작사례 검색 결과에서 보여줘야 할 정보는 무엇인가요?  
    예: 상품 사진, 사양서, 견적서, 제조사, 단가, MOQ, 납기, 이슈, 최종 판매가, 재발주 여부.
71. 과거 사례를 기반으로 예상 원가를 제안할 때 허용 가능한 오차 범위는 어느 정도인가요?
72. AI가 추천한 제조사나 원가를 사람이 수정하면, 그 수정 사유를 기록해야 하나요?
73. 승인된 최종본은 어디에 저장되어야 하나요?  
    ERP, 원드라이브, 신규 시스템, 또는 모두 연동.
74. 승인 완료 후 수정이 발생하면 새 버전으로 관리해야 하나요, 기존 문서를 덮어써도 되나요?
75. MVP에서 가장 먼저 자동화하면 체감 효과가 큰 기능 1개는 무엇인가요?  
    후보: 과거 사례 검색, 작업지시서 초안, 견적 비교, 품질표시사항 초안, 제조사 추천, 일정 지연 알림.

# **핵심 질문 1**

## **“이 문서를 받으면 실제 업무를 하기 위해 반드시 확인하는 정보는 무엇인가요?”**

이 질문이 가장 중요합니다.  
작업지시서와 견적서 양식이 다 다르면, 문서에 적힌 항목명은 믿을 수 없습니다. 대신 업무자가 실제로 확인하는 **필수 의미 정보**를 뽑아야 합니다.

예를 들어 이렇게 물어보면 됩니다.

“작업지시서나 견적서를 받았을 때, 업무를 진행하려면 반드시 봐야 하는 정보가 뭐예요?  
문서마다 표현은 달라도 꼭 확인하는 항목을 순서대로 말해주세요.”

하위 질문은 이렇게 들어가면 됩니다.

1. 이 문서에서 **없으면 일을 진행할 수 없는 정보**는 무엇인가요?  
    예: 상품명, IP명, 캐릭터명, 수량, 사이즈, 소재, 색상, 인쇄 방식, 포장 방식, 납기, 단가, MOQ, 제조사, 담당자, 인증 여부.
2. 있으면 좋지만 없어도 일단 진행 가능한 정보는 무엇인가요?
3. 문서에는 없지만 항상 메일, 메신저, 이전 파일, 담당자 기억으로 보완하는 정보는 무엇인가요?
4. 같은 정보가 문서마다 다르게 표현되는 경우가 있나요?  
    예: “수량”, “발주수량”, “오더수량”, “제작수량”, “입수”, “MOQ”가 섞여 쓰이는지.
5. 작업지시서와 견적서에서 같은 항목이 다르게 적혀 있으면 어느 쪽을 더 믿나요?

이 질문을 통해 얻어야 하는 결과는 이것입니다.

```text
라벨링 대상 필드 후보
- product_name
- ip_name
- character_name
- product_type
- sku_option
- quantity
- unit_price
- moq
- material
- size
- color
- print_method
- package_type
- delivery_date
- vendor_name
- certification_required
- remarks
```

하지만 여기서 끝내면 안 됩니다.  
각 필드는 다시 이렇게 나눠야 합니다.

```text
필수 필드
- 없으면 업무 진행 불가

조건부 필드
- 특정 상품군에서만 필요
예: 전자제품이면 전파인증, 완구면 KC, 봉제인형이면 충전재/겉감/사용연령

참고 필드
- 있으면 좋지만 없어도 진행 가능

외부 참조 필드
- 문서에 없고 메일/메신저/ERP/과거 파일에서 찾아야 하는 정보
```

이 질문의 진짜 목적은 **라벨 이름 정하기**가 아니라,  
“AI가 어떤 정보를 못 찾으면 업무자에게 다시 물어봐야 하는가?”를 정하는 것입니다.

# **핵심 질문 2**

## **“양식은 달라도 같은 의미로 봐야 하는 표현과, 반대로 비슷해 보여도 다르게 봐야 하는 표현은 무엇인가요?”**

이 질문은 온톨로지에서 매우 중요합니다.  
데이터가 제각각이면 AI가 가장 많이 틀리는 부분이 **동의어, 약어, 현장 표현, 문맥 의존 표현**입니다.

업무자에게 이렇게 물어보면 됩니다.

“문서마다 표현은 다른데 실무적으로는 같은 뜻으로 보는 말들이 있나요?  
반대로 비슷해 보이지만 실무적으로는 완전히 다르게 구분해야 하는 말도 있나요?”

하위 질문은 이렇게 하면 됩니다.

1. 같은 의미로 취급하는 표현을 알려주세요.  
    예:
    - 단가 / 공급가 / 제작단가 / 납품가
    - 수량 / 발주수량 / 제작수량 / 오더수량
    - 샘플 / 1차 샘플 / 확인 샘플 / 최종 샘플
    - 패키지 / 포장 / 박스 / OPP / 행택
    - 원단 / 소재 / 겉감 / 충전재
2. 비슷해 보이지만 반드시 구분해야 하는 표현은 무엇인가요?  
    예:
    - 견적 단가 vs 확정 단가
    - 샘플 단가 vs 양산 단가
    - MOQ vs 실제 발주 수량
    - 납기 예정일 vs 입고 예정일 vs 출시 예정일
    - 상품명 vs ERP 품목명 vs 거래처 요청명
    - 공급가 vs 판매가
    - 원가 vs 제조원가 vs 총원가
3. 문서에서 자주 나오는 약어, 줄임말, 내부 용어는 무엇인가요?
4. 업체마다 다르게 쓰는 표현이 있나요?
5. 같은 단어라도 상품군에 따라 의미가 달라지는 경우가 있나요?

이 질문을 통해 라벨링은 단순히 이렇게 하면 안 됩니다.

```text
문서에 "단가"라고 적힌 부분을 unit_price로 라벨링
```

이렇게 해야 합니다.

```text
price_value
price_type:
  - sample_price
  - mass_production_price
  - confirmed_price
  - estimated_price
  - supply_price
  - retail_price
  - total_cost
```

즉, “단가”라는 글자만 라벨링하면 안 되고,  
**그 단가가 어떤 업무 의미의 단가인지**까지 라벨링해야 합니다.

이 질문의 결과물은 아래처럼 정리해야 합니다.

```text
표현 사전 / 동의어 사전 / 금지 혼동표

예시:
- "공급가", "납품가" → supply_price 후보
- "소비자가", "판매가" → retail_price 후보
- "MOQ", "최소수량" → minimum_order_quantity
- "발주수량", "오더수량" → order_quantity
- "샘플비" → sample_cost
- "금형비" → mold_cost
```

이게 없으면 나중에 AI가 견적서를 읽어도  
“금액은 뽑았는데 이게 원가인지, 판매가인지, 샘플비인지 모르는” 문제가 생깁니다.

# **핵심 질문 3**

## **“이 문서를 보고 다음 단계로 넘길지, 다시 확인할지, 위험하다고 판단하는 기준은 무엇인가요?”**

이 질문이 AI 네이티브 온톨로지화의 핵심입니다.  
단순 데이터 라벨링은 “문서 안의 글자 추출”에서 끝납니다.  
하지만 회사 업무에 쓸 AI는 문서를 보고 **다음 액션**을 판단해야 합니다.

업무자에게 이렇게 물어보면 됩니다.

“이 작업지시서나 견적서를 봤을 때, 바로 진행해도 되는지, 다시 물어봐야 하는지, 위험하다고 보는 기준은 뭐예요?”

하위 질문은 이렇게 들어가면 됩니다.

1. 어떤 정보가 누락되면 반드시 다시 물어봐야 하나요?
2. 어떤 값이 이상하면 위험 신호인가요?  
    예:
    - 단가가 과거보다 너무 낮거나 높음
    - MOQ가 비정상적으로 높음
    - 납기가 생산 리드타임보다 짧음
    - KC 인증 필요한 상품인데 인증 항목 없음
    - IP 승인 전인데 양산 일정이 잡힘
    - 최종 샘플 승인 전인데 발주 요청이 들어옴
3. 작업지시서와 견적서 내용이 충돌할 때 어떻게 판단하나요?  
    예: 수량 다름, 소재 다름, 납기 다름, 단가 다름.
4. 어떤 경우에는 팀장, 대표, 영업팀, 디자인팀, 제조사에 반드시 확인해야 하나요?
5. 과거에 실제로 문제가 되었던 누락·오류 사례는 무엇인가요?

이 질문을 통해 만들어야 하는 것은 단순 필드 라벨이 아니라 **검증 규칙 라벨**입니다.

예를 들면 이렇게 됩니다.

```text
문서 라벨링 결과
- product_type: 봉제인형
- quantity: 3,000
- material: 누락
- size: 25cm
- certification_required: 판단 필요
- sample_approval_status: 없음
- production_status: 발주 요청

AI 판단
- missing_required_field: material
- risk_flag: sample_not_approved_before_order
- next_action: 담당자에게 소재와 샘플 승인 여부 확인
```

이 질문의 진짜 목적은 AI에게 이런 능력을 주는 것입니다.

```text
“이 견적서는 단가 추출은 되었지만, MOQ와 실제 발주수량이 다르고,
최종 샘플 승인 여부가 없어 양산 발주 전 확인이 필요합니다.”
```

이게 되어야 업무자가 쓸 수 있는 AI가 됩니다.

# **내가 추천하는 실제 인터뷰 방식**

업무자에게 추상적으로 묻지 말고, 실제 파일 3개를 열어놓고 물어보는 게 좋습니다.

1. 잘 정리된 작업지시서 1개
2. 엉망이지만 실제로 처리했던 작업지시서 1개
3. 견적서 1개

그리고 이렇게 진행하면 됩니다.

```text
“이 파일을 처음 받았다고 생각하고,
업무 진행을 위해 어디부터 보시는지 화면 보면서 말씀해주세요.”
```

그다음 업무자가 보는 순서대로 물어보세요.

```text
“지금 이 항목을 왜 보셨어요?”
“이게 없으면 어떻게 하세요?”
“이 표현은 다른 파일에서는 뭐라고 적혀 있나요?”
“이 값이 이상하면 어떤 문제가 생기나요?”
“이건 누가 최종 확인해야 하나요?”
```

이 방식이 가장 좋습니다.  
왜냐하면 업무자는 “필수 필드를 말해주세요”라고 하면 잘 못 떠올리지만, 실제 문서를 보면서는 자연스럽게 판단 기준을 말해주기 때문입니다.

---

# **결론**

지금 필요한 핵심 질문 3개는 단순히 라벨명을 정하는 질문이 아닙니다.

1. **무엇을 반드시 읽어야 하는가?**
2. **다른 표현들을 어떻게 같은 의미로 묶거나 구분해야 하는가?**
3. **읽은 결과로 어떤 판단과 다음 액션을 하는가?**

이 3개를 뽑으면, 아이콘스의 작업지시서·견적서 라벨링은 단순 문서 정리가 아니라  
**AI가 상품 개발 업무를 이해하는 온톨로지의 기초 데이터셋**이 됩니다.
