## 컨텍스트

이 폴더에 있는 기존 분석 및 기획 문서들을 모두 읽어줘:

**우리 앱 기획 문서:**

- PRD.md
- ARCHITECTURE.md
- IMPLEMENTATION-PLAN.md
- user-stories.md

## 회의 원본

아래는 MVP 스코프 회의에서 나온 내용을 정리 안 된 상태로 옮긴 거야. 두서없을 수 있으니까 네가 구조화해줘.

```
탭바 순서: 홈 스토어 디엠(mvp에서는 누를 시 곧 출시예정이라는 화면과 함께 알람받기 버튼이 있음) 커밍순 더보기

아티스트가 그룹일때 멤버리스트 보여주고 각 멤버마다 프로필 보기 버튼이 있다. 각 멤버별로 팔로잉 기능있고 그 멤버의 프로필에서 게시글, 댓글만 모아볼 수 있다. 팔로잉을했을때 그 멤버의 피드 및 디엠 알람이 온다

아티스트가 솔로일땐 똑같지만 혼자만 보인다

앱 실행시 스플래쉬 화면 나오고 로그인화면이 나온다. 앱을 처음 다운받은 사용자에 한해서만 스포티파이 시작시 큐레이션 선택하기 처럼 아티스트들의 리스트가 전체 나온다 (아직 알고리즘은 따로 없고 랜덤으로) (건너뛰기가능하고 건너뛰기하면 홈화면으로 돌아가며 홈화면의 아티클에서 아티스트 추천이 나옴 이것도 랜덤이다)
```

## 작업 지시

### Step 1: 회의 내용 구조화

회의 원본을 읽고 아래 4가지로 분류해서 `meeting-summary.md`로 저장해줘.

**1) 확정 사항** — 명확하게 결정된 것

- 기능명
- MVP 포함 여부 (MVP / v1.1 / v2.0)
- 핵심 요구사항 요약

**2) 제안/논의 사항** — 아이디어로 나왔지만 확정은 안 된 것

- 내용
- 찬반 의견이 있었다면 정리

**3) 삭제/후순위 결정** — MVP에서 빼기로 한 것

- 기능명
- 이유
- 어느 버전으로 미뤘는지

**4) 미결 이슈 / 추가 확인 필요** — 결정 못 한 것, 모호한 것

- 내용
- 결정을 위해 필요한 추가 정보

⚠️ 내가 해석이 모호하거나 확신이 안 되는 부분은 `[확인 필요]` 태그를 달아줘. 원본 메모에서 의미를 추측해서 임의로 확정하지 마.

**이 단계를 나에게 보여주고 확인받기 전까지 다음 단계로 넘어가지 마.**

---

### Step 2: 변경 영향도 분석

내가 Step 1을 확인해주면, 확정 사항만 대상으로 기존 문서에 미치는 영향을 분석해줘. `change-impact.md`로 저장.

각 변경사항마다 아래 형식으로 작성:

```
## 변경 #N: [기능명/변경 내용]

### PRD 영향
- 수정 섹션: [섹션명]
- 변경 내용: [추가/수정/삭제할 내용 요약]
- 우선순위 제안: [P0/P1/P2]

### ARCHITECTURE 영향
- 데이터 모델: [새 테이블/컬럼 추가 여부]
- API: [새 엔드포인트 필요 여부]
- 외부 서비스: [새로운 의존성 여부]
- 기존 컴포넌트 변경: [영향받는 모듈]

### IMPLEMENTATION-PLAN 영향
- 예상 추가 공수: [N일]
- 배치할 스프린트: [스프린트 N]
- 선행 의존성: [먼저 완료되어야 할 기능]
- 기존 일정 밀림 여부: [있음/없음, 있으면 상세]
  
  ## user-stories.md 영향
```

마지막에 전체 영향도 요약을 추가해줘:

- 총 추가 공수 예상
- 기존 MVP 일정에 미치는 영향
- 기술적 리스크가 높은 변경사항

**이 단계도 나에게 보여주고 확인받기 전까지 다음 단계로 넘어가지 마.**

---

### Step 3: 문서 업데이트

내가 Step 2를 확인해주면 기존 문서를 업데이트해줘.

**백업 먼저:**

- 기존 PRD.md → archive/PRD_v1.md
- 기존 ARCHITECTURE.md → archive/ARCHITECTURE_v1.md
- 기존 IMPLEMENTATION-PLAN.md → archive/IMPLEMENTATION-PLAN_v1.md
- 기존 user-stories.md → archive/user-stories_v1.md


**업데이트 규칙:**

- 새로 추가/변경된 모든 부분에 `<!-- 회의 반영 YYYY-MM-DD -->` 주석 추가
- 삭제하지 않고 후순위로 미룬 기능은 취소선 대신 별도 "Post-MVP" 섹션으로 이동
- 유저스토리가 필요한 새 기능은 기존 넘버링 이어서 US-XXX로 추가
- ARCHITECTURE의 Mermaid 다이어그램도 새 컴포넌트 반영해서 업데이트
- IMPLEMENTATION-PLAN의 스프린트 타임라인 재조정

**새로 생성할 문서:**

- `CHANGELOG.md` — 이번 변경사항 이력 (향후 회의 반영 때도 여기에 누적)

---

### Step 4: 정합성 검증

업데이트 완료 후 문서 간 정합성을 검증하고 결과를 나에게 보고해줘.

체크 항목:

- [ ] PRD의 모든 MVP 기능이 ARCHITECTURE에 기술적으로 반영되어 있는지
- [ ] PRD의 모든 MVP 기능이 IMPLEMENTATION-PLAN 스프린트에 배치되어 있는지
- [ ] ARCHITECTURE의 데이터 모델이 모든 새 기능을 커버하는지
- [ ] ARCHITECTURE의 API 설계가 모든 새 기능의 엔드포인트를 포함하는지
- [ ] 스프린트 간 의존성에 모순이 없는지
- [ ] Weverse 분석 문서에서 참조한 인사이트가 여전히 유효한지

정합성 이슈가 발견되면 `REVIEW-NEEDED.md`에 이슈별로 정리하고, 각 이슈에 대한 해결 제안도 함께 달아줘.

---

## 주의사항

- 각 Step은 반드시 내 확인을 받은 후에 다음으로 진행
- 추측은 `[추정]`, 확인 필요한 것은 `[확인 필요]`로 명확히 표기
- 기존 문서의 확정된 기술 스택 선택은 변경하지 마 (회의에서 명시적으로 바꾸자고 한 경우만 예외)
- 회의에서 언급 안 된 기존 기능은 건드리지 마
